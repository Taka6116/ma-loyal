# 記事導入文の多様性設計書 v1.0

## 1. 現状の問題

### 1.1 症状
どの記事を書いても、冒頭が以下のような文体に収束する：

> 「後継者が見つからないまま、気がつけば自分も60歳を超えていた。売上は安定しているのに、
> このまま事業を続けていけるのか、10年後の会社の姿が描けない。」

読者にとって「また同じ入り方だ」と見えた時点で、記事の信頼性と独自性が損なわれる。

### 1.2 原因（3層構造）

**第1層：例文がアンカーになっている**
プロンプト内に固定の例文が2つあり、LLMはこれを「正解」として模倣する。
どんなパターン指定をしても、最終出力が例文の語彙・構造に引っ張られる。

**第2層：パターンリストが同質的**
8パターン中5つが「経営者の不安・悩み」系。実質3パターン程度しかない。
しかもパターンは「〜から始める」という抽象的な指示なので、AIは結局デフォルトの例文風に書く。

**第3層：「悩みから始めろ」という絶対ルール**
すべての記事が「悩み→共感→本題」の構造を取る。
記事のテーマが違っても、入口が同じ構造なので読者には同じに見える。

---

## 2. 設計思想

### 2.1 パターン化をやめる

「8パターンから1つ選ぶ」方式は廃止する。パターンリスト自体が多様性の上限を固定してしまう。

代わりに、AIに**「この記事のテーマ・キーワード・想定読者から逆算して、最も引き込める導入の一文目を自分で設計しろ」**と指示する。

### 2.2 例文を消す

固定の例文は一切プロンプトに入れない。例文があるとAIは自分で考えることをやめる。

### 2.3 「やっていいこと」ではなく「やってはいけないこと」で制御する

多様性を生むのは自由度。自由度を担保しつつ品質を保つには、
「こう書け」（ポジティブ制約）ではなく「これだけはやるな」（ネガティブ制約）が有効。

### 2.4 過去の導入文を「使用禁止フレーズ」として渡す

物理的に重複を排除する。直近5〜10記事の導入一文目をシステムが記憶し、
「以下のフレーズ・構造は今回使用禁止」としてプロンプトに注入する。

## 3. 実装設計

### 3.1 変更ファイル

`src/lib/api/gemini.ts` のみ

### 3.2 削除するもの

- `INTRO_OPENING_PATTERNS` 定数（8パターンのリスト）
- `pickRandomIntroOpeningPattern()` 関数
- プロンプト内の `❶ 冒頭は読者の悩みから始める` セクション全体（例文2つ含む）
- `今回の冒頭パターン指定：${introOpeningPattern}` の行

### 3.3 追加するもの

#### A. プロンプト内の新しい ❶ セクション

```
❶ 導入（冒頭の2〜3文）の設計
■ あなたの仕事：
今回の記事テーマ・キーワード・想定読者（50〜70代の中小企業経営者）を踏まえ、
「この記事でしか使えない、この記事だからこそ意味がある一文目」を自分で考えること。

■ 導入で禁止すること（以下に該当する書き出しは不可）：
・「近年〜」「〜が問題となっています」「〜が注目されています」などの時事導入
・「後継者が見つからないまま」「気がつけば●歳を超えていた」のような年齢+後継者不在の定型描写
・「売上は安定しているのに」「会社の将来が描けない」のような漠然とした将来不安の描写
・前回までの記事と同じ構造・同じ語彙の導入（下記の使用禁止リスト参照）
・「ある日突然〜」のような物語調の紋切り型

■ 導入で意識すること：
・読者が「お、これは今まで読んだ記事と違うな」と感じる一文目にする
・テーマに固有の具体的な場面・数字・問い・事実から入る
・「悩みの共感」以外にも、反直感的な事実提示、具体的なシーン描写、
  読者への直接的な問いかけ、業界の構造的な変化の指摘など、自由にアプローチしてよい

${usedIntrosBlock}
```

#### B. `usedIntrosBlock` の生成ロジック

過去の記事の導入一文目をlocalStorageまたはAPI経由で取得し、以下の形式でプロンプトに注入する。

```
■ 使用禁止：過去の記事で使った導入（類似の構造・語彙も不可）
1.「後継者が見つからないまま、気がつけば自分も60歳を超えていた。」
2.「M&Aという言葉に、かつてほどの抵抗はなくなった。」
3.「従業員30名の製造業を経営するTさん（仮名）は、ある朝〜」
※上記と似た語彙・構造・トーンの書き出しは使用しないこと。
```

過去データがない場合（初回等）は、このブロック自体を省略する。

#### C. データの保存

**方針：サーバー側で保存**

記事の一次執筆完了時に、生成された本文の先頭1文（句点まで）を抽出し、
`data/intro-history.json` に最大20件まで保存する（FIFO）。

```typescript
interface IntroHistoryEntry {
  date: string       // ISO 8601
  keyword: string    // ターゲットキーワード
  firstSentence: string  // 導入の一文目
}
```

### 3.4 推敲プロンプトの変更

`refineArticleWithGemini` 内の `【B. 冒頭】` セクションも更新する：

変更前：
```
□ 「近年〜」「〜が問題となっています」で始まっていないか
  → もし始まっていたら「読者の悩みの言語化」に書き直す
□ 50〜70代経営者が「自分のことだ」と感じる描写になっているか
```

変更後：
```
□ 「近年〜」「〜が問題となっています」「〜が注目されています」で始まっていないか
□ 「後継者が見つからない」「気がつけば●歳」「売上は安定しているのに」などの
  定型フレーズで始まっていないか
  → もし該当していたら、この記事のテーマに固有の具体的な場面・事実・問いから始まる
  導入に書き直す
□ 50〜70代経営者にとって「読んだことがない切り口だ」と感じられる導入になっているか
```

---

## 4. 変更の影響範囲

| 項目 | 影響 |
|------|------|
| 一次執筆の出力 | 導入が毎回異なるものになる |
| 推敲の出力 | 定型導入を検出して書き直す |
| 既存の記事 | 影響なし |
| エディタUI | 変更なし |
| API Route | 変更なし |
| ビルド | TS変更のみ、型変更なし |

---

## 5. 期待される効果

| 改善前 | 改善後 |
|--------|--------|
| 毎回「後継者が〜65歳〜売上は安定〜」 | テーマごとに異なる切り口の導入 |
| 8パターンから選ぶ → 実質3パターン | パターンリスト廃止 → AIが毎回テーマから逆算 |
| 例文がアンカー → 模倣 | 例文なし + 禁止リスト → 創造を強制 |
| 過去記事との重複チェックなし | 過去20件の導入を使用禁止として注入 |
